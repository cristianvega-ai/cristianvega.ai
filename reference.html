DECISION 01

Repository strategy

The real question isn't "monorepo or not" — it's where to draw the seams. There's a spectrum, and both extremes are wrong for a team your size.

The repository spectrum — and where you sit
Giant monorepo
Everything in one repo. Google / Meta scale — needs Bazel-class tooling.
Repos by domain
A handful of repos, each a bounded context. Where you're headed.
Repo per app
One repo per deployable. Clear ownership, heavy coordination cost.
Repo per service
Full micro-repo sprawl. Amazon/Netflix at scale — drift & overhead for small teams.
◆ Your 3-repo plan lands here — the pragmatic middle most mid-size orgs converge on

Monorepo vs. many repos — the honest trade

"Monorepo" just means related code shares a repo and a toolchain; it does not mean one giant deployable. Here's what each side actually buys you.

MFewer, larger repos (monorepo-leaning)
  • Atomic cross-app changes. Change a shared contract and every consumer in one commit — no coordinated multi-repo PR dance.
  • Effortless code sharing. Types, UI components, and the Okta auth wrapper are imported directly, always in sync.
  • One toolchain. One lockfile, one lint/format/CI config, one "version of everything" policy. Less drift.
  • Better for AI-assisted work. This is your stated reason for Crimson+Ember, and it's legitimate — agents assemble context far more easily when related code is co-located.
  • Discoverability. New engineers (and models) see the whole picture in one place.
MCosts of going too broad
  • CI needs real tooling. Without affected-graph builds (Nx, Turborepo, Bazel), every change rebuilds everything and CI grinds to a halt.
  • Coarser access control. Everyone with repo access sees all of it — fine internally, a problem if a security boundary runs through it.
  • Coupling creep. "Easy to share" quietly becomes "everything depends on everything" without discipline about boundaries.
  • Checkout weight. Grows over time; matters at large scale, not yet at yours.
PMore, smaller repos (polyrepo-leaning)
  • Hard ownership boundaries. Repo = team = deploy unit = access scope. Clean and obvious.
  • Independent lifecycles. Reporting can move at its own cadence without touching the product suite.
  • Fine-grained access. Grant repo-by-repo — valuable across a customer-facing / internal split.
  • Contained blast radius. A bad change can't ripple beyond its repo.
PCosts of going too narrow
  • Sharing gets bureaucratic. Shared code must be published to a private registry and versioned — every bump is a release + PR in each consumer.
  • Dependency drift & diamond problems. Repos pin different versions; upgrading a shared lib across all of them becomes a project.
  • Duplicated everything. CI config, lint rules, auth setup, tooling — copied and slowly diverging.
  • Worse AI/agent context. The exact friction you removed for Crimson+Ember, reintroduced.
The principle to internalize

A repo is a code-organization choice. A deploy unit is a runtime choice. They're independent.

You can have 1 repo → N independently deployed apps (exactly what Crimson + Ember are today), or N repos → N apps, or 1 repo → 1 monolith. Conflating "same repo" with "ships together" is the single most common source of the confusion you're describing. Decide code layout for developer and AI ergonomics; decide deploy units for runtime independence. Keep them in separate mental columns.

Recommendation

Ship your three-repo plan — with one addition and one rule

Grouping by bounded context (Product Suite / Internal Tools / Reporting) is the right call at your scale. It gives you clean ownership and a natural place for the likely customer-facing vs. internal security boundary, while keeping each repo small enough that AI-assisted work stays effective. Two things make it hold up:

  • Add a shared-platform package. All apps need Okta login, a design system, and common types. Publish these once — as a workspace package or a small private package — instead of reimplementing auth in three repos and watching them diverge.
  • Adopt a "new repo" bar. Open a repo only for a genuine new bounded context, a distinct security domain, a separate owning team, or a truly independent lifecycle. "A team would like their own" isn't sufficient on its own — reporting qualifies because it is a distinct analytics domain, not merely a preference.

If your team is small (say, under ~8 engineers) and everything is internal, a single monorepo with three app folders is also completely valid and even simpler — the AI-context and code-sharing wins compound. The deciding factor is whether you need the security/ownership boundaries the three-repo split gives you.

DECISION 02

Deployment model

You've already got the hard part right. The goal is to make "independent deployability" the default for every app and let CI stay fast as the repos grow.

The modern baseline
unitApp = deploy unit. Each app builds and ships on its own, whatever repo it lives in. Crimson and Ember already work this way.
buildBuild only what changed. Path-based CI triggers (or an affected-graph tool) so a tweak to one app doesn't rebuild the whole repo.
packageContainerize. A Docker image per app is the portable, boring, correct default — runs the same everywhere.
infraInfrastructure as code. Terraform / Pulumi / CDK so environments are reproducible and reviewable, never click-ops.
promoteDev → staging → prod with preview environments per pull request. Roll forward fast; roll back faster.
Where things run (pick per app type)
frontendsVercel / Netflix-style edge or Cloudflare / S3+CDN. Static and SSR frontends deploy in seconds with preview URLs built in.
servicesCloud Run / AWS App Runner / ECS Fargate. Container-native, scale-to-zero, minimal ops — ideal until you truly need more.
at scaleKubernetes (EKS/GKE). Powerful and standard, but real operational weight. Adopt when you've outgrown the managed options — not before.
Don't over-provision the platform. A five-app portfolio does not need a service mesh. Start with the simplest thing that deploys independently and graduate deliberately.
Recommendation

Keep the Crimson/Ember pattern; make it the standard everywhere

Formalize what you're already doing: one CI/CD pipeline per repo that detects which app(s) changed and deploys only those, each as an independent container to a managed runtime. Manage infrastructure as code from day one. This keeps repo structure and deployment fully decoupled — you can reshuffle repos later without re-architecting how anything runs.

DECISION 03

Identity, Okta & navigation

This is where you and engineering disagree — and the disagreement dissolves once you separate three things that "Okta tiles" quietly bundles together: authentication, authorization, and discovery.

The unlock

Three different jobs, often mistaken for one

Authentication — proving who you are. This is Okta's job, done once via SSO. Non-negotiable, and both plans agree here.

Authorization — what you're allowed to do. Split it: coarse "can you open this app at all" fits Okta groups; fine "what can you see and do inside" belongs to the app.

Discovery / navigation — how you find and move between apps. Pure UX. A wall of Okta tiles is one way to do it, not the only way — and not a great one past a handful of apps.

The two plans, stated fairly

Plan A
A tile per app in Okta
// engineering's instinct
+Dead simple — nothing to build or maintain. Okta is the whole front door.
+Users see exactly what they can access; deep links work naturally.
+No shared shell means no single point of failure across apps.
−A wall of tiles becomes noise fast; no shared nav or consistent experience.
−Tempts teams to push routing and permissions into Okta, which it isn't built for.
Plan B
One entry → app shell → route inside
// your instinct
+A launchpad gives cohesive suites shared nav and one consistent experience.
+Routing and permissions live in the app, where the context is — exactly right.
+Mirrors how big suites actually work (the Google "waffle," Microsoft 365 launcher).
−The shell is real software to build and own, and can be a single point of failure if careless.
−Overkill if applied to genuinely standalone apps that share nothing.
Recommendation

It's not either/or — pick per suite, on one identity backbone

You're right on the point you care about most: in-app routing and permissions should not live in Okta. Engineering is right that Okta should be the front door for authentication. Both hold at once:

  • Okta SSO for every app, always. Each app is an OIDC relying party. One login. This never changes regardless of navigation style.
  • Coarse access via Okta groups → token claims. Group membership decides who can open the product suite / internal tools / reporting at all, and rides along in the token.
  • Fine access via in-app RBAC. Once inside, the app reads those claims and decides what the user sees and does. This is the "we choose where you go inside the app, not Okta" you described.
  • A launchpad for cohesive suites; tiles for loners. Give the product suite (and later Internal Tools) a single entry with an in-app switcher. Leave a genuinely standalone reporting app as its own tile. Keep the Okta dashboard to a few meaningful tiles, not one per micro-app.
  • Share the Okta session across a suite. Put a suite's apps under one domain (subdomains or paths). The user authenticates once and moves between apps with no re-login — the seamless feel you're after.

Net: authentication is centralized and boring; authorization is layered (coarse in Okta, fine in-app); navigation is chosen per suite based on whether the apps actually belong together.

BENCHMARK

What big tech actually chooses

The honest headline: there's no single "correct" answer, and every large org is really a mix. But the leanings are real, and they track two things — engineering scale and tooling investment.

CompanyCode organizationDeploymentIdentity & navigation pattern
Google Monorepo One vast repo on custom tooling (Piper + Bazel). Independent services from a shared repo, built with an affected-graph system. Central SSO; the "waffle" app switcher unifies a suite of apps behind one login.
Meta Monorepo Large monorepos with purpose-built VCS & build (Sapling, Buck2). Heavy internal CD tooling; independent deployables. Central SSO across internal tools.
Microsoft Mixed Giant monorepos (Windows/Office, Git-at-scale) and many polyrepos. Azure DevOps / GitHub Actions pipelines per product. Entra ID (Azure AD) SSO; the Microsoft 365 launcher is the canonical app-switcher.
Amazon Polyrepo Thousands of small packages; two-pizza teams own their own. Per-service pipelines (Brazil build); services deploy fully independently. Central SSO; access scoped per service/team.
Netflix Polyrepo Many repos aligned to microservices & team autonomy. Spinnaker-driven continuous delivery per service. Central SSO across independent services.
Uber Monorepo Language monorepos (Go, iOS, Android) on Bazel. Independent service deploys from shared monorepos. Central SSO; internal platform portals for tooling.
Atlassian Mixed Consolidating toward larger repos over time. Per-product delivery pipelines. Textbook launchpad: one identity, a product switcher across the suite.

Pattern to take away: Google, Meta, and Uber lean monorepo; Amazon and Netflix lean polyrepo — and the split tracks scale and tooling, not correctness. On identity, the constant across all of them is one central SSO, with a launchpad/switcher wherever apps form a coherent suite. Your three-repo, Okta-SSO, launchpad-per-suite plan is squarely inside the mainstream.

GUARDRAILS

Traps to design around

The failure modes that turn a reasonable plan into a painful one. None are hypothetical — each is a common way portfolios like yours go sideways.

Repo sprawl by preference
Opening a repo whenever a team "wants one" ends in a dozen repos with drifting tooling and no clear map. Gate new repos behind a real bounded-context or security reason.
Conflating repo with deploy
Assuming "same repo means ships together." It doesn't. Keep code layout and runtime units as separate decisions — you already do this well; don't lose it.
Okta as a routing engine
Pushing fine-grained, in-app permissions and navigation into Okta groups. Okta authenticates and gates app-level access; the app owns everything past the front door.
Reinventing auth per repo
Three repos each hand-rolling Okta login guarantees three subtly different, subtly broken implementations. Centralize it in the shared platform package.
Over-building the platform
Kubernetes, a service mesh, and micro-frontends for five apps and a small team. Buy that complexity only when real scale forces it.
Monorepo without affected-builds
If you do consolidate, skipping path filters or an affected-graph tool means CI rebuilds everything on every change and slows to a crawl. Tooling is the price of the monorepo ticket.
REUSABLE RULE

When to open a new repo

So the next "should this be its own repo?" answers itself. Same repo unless a split earns its keep.

Keep it in an existing repo when…

  • ✓It's the same bounded context / product line.
  • ✓It shares real code — types, UI, auth — with what's there.
  • ✓It tends to change alongside its neighbours.
  • ✓The same team owns it.
  • ✓It has the same security posture as its repo-mates.

Split into a new repo when…

  • ✓It's a genuinely distinct bounded context (reporting qualifies).
  • ✓It's a different security or compliance domain (customer-facing vs. internal).
  • ✓A separate team needs real autonomy over it.
  • ✓Its release lifecycle is truly independent.
  • ✓You need access control it can't get sharing a repo.
TO SHARPEN THIS

Five answers that would tighten the recommendation

This brief runs on a few reasonable assumptions. Confirm or correct these and I'll turn the general framework into a concrete, wired-up plan for your exact stack.

Q1
Is the product suite customer-facing, or all internal?
This is the pivotal one. A customer-facing / internal boundary is the strongest argument for keeping the suite in its own repo with its own security posture — and changes how hard the Okta group split needs to be.
Q2
What's the stack, and how uniform is it?
All TypeScript points to Turborepo or Nx and effortless package sharing. Polyglot (say, Python services + JS frontends) shifts the calculus toward looser coupling or Bazel-class tooling.
Q3
How many engineers, and how are they grouped?
Under ~8 on one team makes a single monorepo genuinely attractive. Multiple autonomous teams tilt toward the repo-per-domain split you proposed.
Q4
Where do you deploy today?
AWS, GCP, Azure, or a PaaS like Vercel — this determines the concrete runtime picks (Cloud Run vs. ECS vs. edge) and the IaC tool that fits.
Q5
Do the suites share a UI, or just a login?
A shared shell and design language justifies a real launchpad (and possibly micro-frontends). If apps only share authentication, a lightweight portal plus per-app tiles is plenty.

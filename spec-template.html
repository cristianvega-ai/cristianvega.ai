1.1 · Summary

What this is

When a run fails partway through a workflow, operators today re-key the whole thing from scratch. This spec adds first-class retry: resume a failed run from its last good checkpoint, skipping the blocks that already completed, with a truthful audit trail linking the original run to its recovery.

Why now: failure recovery is the top operator complaint on the runs surface, and the engine already checkpoints every block boundary, so the expensive half of the problem is already solved.

1.2 · Problem & context

The problem

A seven-block workflow that fails at block 3 discards the work of blocks 1 and 2. The operator's only recovery is to start a brand-new run and re-enter every input, and nothing in the history connects the failed attempt to its replacement. On long document workflows this turns a transient timeout into many minutes of duplicate work, and it quietly corrupts our reliability picture: every recovered failure looks like two unrelated runs.

Plain english

Today a failed run is a torn-up form: you copy everything onto a fresh one by hand. Retry is "resume the download" instead of "download it again."

1.4 · Goals · Non-goals

What we're signing up for, and what we're not

Goals
  • Recover a failed run without re-keying inputs or re-executing completed blocks.
  • Keep the audit trail truthful: parent and child runs are linked and both tell the whole story.
  • Ship behind a flag, reversible with one switch and no data cleanup.
Non-goals · v1
  • Bulk retry from the runs list. Needs a selection model and rate design; parked as a follow-up.
  • Editing a run's inputs before resuming. Retry replays the same run, it does not fork it.
  • Any change to bot scheduling, hosts, or handler packages.
1.5 · Users & primary scenario

Who this is for

Primary · acts

Operator

Works the run queue and is first on the scene when a run fails. Wants recovery in two clicks, from the page they're already on.

Secondary · observes

Builder

Designs the workflow. Doesn't retry runs, but needs failures to stay debuggable: a retry must show exactly which blocks were skipped and why.

  1. A run fails mid-workflow

    Run #4231 fails at block 3 of 7 on an external-write timeout. The operator opens the run detail.

  2. Operator retries from the failure

    They hit Retry from failed block; a confirm strip states exactly what will replay and what will be skipped.

  3. Engine resumes from checkpoint

    A child run is queued starting at block 3. Blocks 1 and 2 are not re-executed.

  4. History stays honest

    The child completes; both runs display the parent/child link, and the failed parent reads "recovered by #4232".

2.2 · Proposed design

Shape of the change

Retry is a thin new path over machinery that already exists. A new engine endpoint validates that the source run is failed, loads its last good checkpoint, and queues an ordinary child run that begins at the failed block. The child is a first-class run: same execution path, same caps, same audit surface, plus a parent_run_id pointer. The console adds one action and one banner; it learns nothing about checkpoints.

2.3 · Key decisions

Forks in the road

DEC-01

Resume from checkpoint, not full re-run

Decided
Choice
Child run resumes at the failed block from the LangGraph checkpoint.
Alternatives
(a) Full re-run with copied inputs: simpler, but re-executes side-effecting blocks. (b) In-place restart of the same run row: no duplicate row, but rewrites history and breaks the audit story.
Why
Checkpoints already exist at block boundaries; resume is the only option that is both cheap and honest.
Revisit when
A run predates checkpointing or its checkpoint fails to load; those fall back to full re-run behind the same button.
DEC-02

Recovery is a new linked run, not a mutation

Decided
Choice
The failed run stays failed forever; recovery is a child run pointing at it.
Alternatives
Flipping the failed run back to running: fewer rows, but every consumer of run history (insights, billing, audit) would need to understand resurrection.
Why
Append-only history keeps all downstream readers correct with zero changes, and the reliability picture stays truthful.
Revisit when
Child-run volume ever distorts run-count metrics; then teach insights to fold chains, not the engine to mutate.

2.5 · Security & tenancy

Stances

Authentication

Unchanged: console BFF session, engine Bearer JWT. No new principals.

Authorization

Retry requires operator persona or above; denials return 403 and are written to the audit log (REQ-03).

Tenancy

The child run inherits the parent's client scope; RLS policies apply as-is. No new tables, so no new policies.

Data handling

No new PII. Checkpoint payloads already pass through the redacting serializer; retry adds no new readers of raw state.

Abuse & limits

Child runs count against the existing per-client concurrent-run cap; one live child per parent at a time.

3.3 · Rollout & backout

Switches

Flag

automation.run_retry, engine-side, default off. Console hides the action when the capability is absent.

Migration

Forward-safe nullable column; runs with no parent are untouched. No backfill, no downtime.

Backout

Flag off. Existing child runs remain valid ordinary runs in history; nothing to clean up.

Comms

Release note plus the operator guide from T3.3; support told which tenants have it on.

4.2 · Interaction notes

What the pins mean

  1. Visible only when the run status is failed and the viewer's persona is operator or above. One live child per parent: while a child is queued or running, the button reads "Retry in progress" and is disabled.
  2. Confirm is an inline strip, not a modal: it states exactly what replays and what is skipped. Enter confirms, Esc cancels, focus lands on Confirm.
  3. After the child completes, the parent gains this banner permanently; the child shows the mirror banner linking back. History reads truthfully in both directions.
States

Empty: never; the surface only exists on a failed run. Loading: the button shows a spinner until the 202 returns, then navigates to the child. Error: 409 (already recovered) refreshes the page state; anything else shows the standard error strip with the audit id.
